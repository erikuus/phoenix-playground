This section explains the rationale behind refactoring our pagination implementation into a helper module. If you haven't reviewed the initial pagination implementation in the previous section, it's recommended to start there to understand the complete context and functionality of the code being refactored.

Key Principles Behind the Helper

  • Reusable across all paginated LiveViews
  • Consistent pagination behavior
  • Encapsulated pagination logic
  • Centralized configuration
  • Clear separation from LiveView lifecycle

The helper handles all pagination logic, letting the LiveView focus on:

  • LiveView lifecycles (mount, handle_params, etc.)
  • Languages feature business logic
  • View-specific state
  • View-specific events

convert_params, to_integer

In helper because:

  • Pure parameter conversion utility
  • No LiveView-specific logic
  • Reusable across paginated views

validate_options

In helper because:

  • Handles all pagination validation
  • Pure computation, no side effects
  • Core pagination rules needed app-wide

get_per_page_select_options

In helper because:

  • Central pagination UI config
  • Consistent options across views
  • Single source for per-page options

update_per_page_option

In helper because:

  • Standard per-page updates
  • Consistent default handling
  • Reusable pagination logic

init

In helper because:

  • Standard pagination setup
  • Handles all pagination assignments
  • Consistent across LiveViews

apply_options

In helper because:

  • Complex, reusable pagination logic
  • Consistent state updates
  • Handles stream resets and URL updates

handle_created, handle_updated, handle_deleted

In helper because:

  • Standard count updates
  • Consistent stream handling
  • Manages pagination state changes

get_summary

In helper because:

  • Pure presentation logic
  • Consistent across views
  • Reusable summary formatting
