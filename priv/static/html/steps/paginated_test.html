Test Cases for Additions

Test Case 1: Adding a New Item to the First Page

  • Scenario: Add a new item while viewing the first page of the list.
  • Steps:
    1. Navigate to the first page.
    2. Add a new item.
  • Expected Outcome:
    • The new item is displayed at the top of the list with a green background and a “New” badge.
    • The summary updates immediately (e.g., “1 - 6 of 101” if 5 items were visible before).
    • Pagination remains unchanged (e.g., still showing 5 pages if applicable).

Test Case 2: Adding a New Item That Belongs to Another Page

  • Scenario: Add a new item that should naturally be sorted into a different page.
  • Steps:
    1. Navigate to any page except the one where the item would belong (based on sorting).
    2. Add a new item.
  • Expected Outcome:
    • The new item is displayed at the top of the current page with a green background and a “New” badge.
    • The item is sorted into its correct position after refresh or navigation.
    • The summary reflects the addition immediately.

Test Cases for Deletions

Test Case 3: Deleting a Single Item from a Full Page

  • Scenario: Delete a single item from a page with the maximum number of visible rows (e.g., 20 items per page).
  • Steps:
    1. Navigate to a page with 20 items.
    2. Delete one item.
  • Expected Outcome:
    • The deleted item is marked with a gray background and a “Deleted” badge.
    • The summary remains gray to indicate a mismatch until refresh.
    • Pagination remains unchanged until navigation or refresh.

Test Case 4: Deleting All Items on a Page

  • Scenario: Delete all items from a single page.
  • Steps:
    1. Navigate to a page with items.
    2. Delete all items on that page.
  • Expected Outcome:
    • All items are marked as deleted.
    • Pagination still displays the same number of pages until refresh or navigation.
    • After refresh, items are fully removed, and the pagination recalculates.

Test Case 5: Deleting All Items on the Penultimate Page

  • Scenario: Delete all items on the second-to-last page.
  • Steps:
    1. Navigate to the penultimate page (e.g., page 5 of 6).
    2. Delete all items on that page.
  • Expected Outcome:
    • After deletion: Pagination still shows page 6, with items in page 5 marked as deleted.
    • After refresh: The single item from page 6 moves to page 5, which becomes the last page.

Test Case 6: Deleting the Only Item on the Last Page

  • Scenario: Delete the only item on the last page.
  • Steps:
    1. Navigate to the last page with only one item.
    2. Delete the item.
  • Expected Outcome:
    • After deletion: Pagination still shows the last page, with the item marked as deleted.
    • After refresh: The user is redirected to the penultimate page, which becomes the last page.

Test Cases for Pagination

Test Case 7: Navigating to a Nonexistent Last Page After Deletion

  • Scenario: Delete all items on the penultimate page and attempt to navigate to the now-nonexistent last page.
  • Steps:
    1. Navigate to the penultimate page (e.g., page 5 of 6).
    2. Delete all items on that page.
    3. Attempt to navigate to page 6.
  • Expected Outcome:
    • The user is redirected to the last valid page (e.g., page 5).
    • After refresh, pagination recalculates, and the last page is updated.

Test Case 8: Refreshing After a Stream Reset

  • Scenario: Refresh the page after multiple additions and deletions.
  • Steps:
    1. Perform several additions and deletions.
    2. Refresh the page.
  • Expected Outcome:
    • All visible items are correctly sorted and paginated.
    • Deleted items are fully removed.
    • Pagination and summary align with the total number of items.

Test Case 9: Manually Manipulating the URL for a Nonexistent Page

  • Scenario: A user manipulates the URL to navigate to a nonexistent page.
  • Steps:
    1. Manually change the page number in the URL to an invalid page (e.g., page 100 when there are only 10 pages).
    2. Load the page.
  • Expected Outcome:
    • The user is redirected to the last valid page (e.g., page 10).
    • Pagination and summary reflect the correct number of items and pages.

Test Case 10: Manually Changing Per-Page Value in the URL

  • Scenario: A user manipulates the URL to set an invalid or extreme per-page value.
  • Steps:
    1. Manually change the per_page parameter in the URL to an invalid or extreme value (e.g., per_page=1000 or per_page=-1).
    2. Load the page.
  • Expected Outcome:
    • The per_page value is validated and set to the nearest valid option (e.g., 20, 50).
    • Pagination and summary are recalculated accordingly.

Test Cases for Real-Time Updates

Test Case 11: Real-Time Addition by Another User

  • Scenario: Another user adds a new item while you’re viewing the list.
  • Steps:
    1. Stay on a page.
    2. Simulate another user adding a new item.
  • Expected Outcome:
    • The new item is displayed at the top of the page with a green background and a “New” badge.
    • A message appears: “A new item has been added by another user.”
    • Pagination remains unchanged until refresh.

Test Case 12: Stream Inserted Times for Additions

  • Scenario: Verify that only added, deleted, or edited rows are updated in the stream.
  • Steps:
    1. Add a new item to the list.
    2. Delete an item from the list.
    3. Edit an item in the list.
  • Expected Outcome:
    • The stream-inserted timestamp for rows reflects updates only for the affected items.
    • Other rows retain their original timestamps, ensuring efficient DOM updates.

Test Case 13: Real-Time Deletion by Another User

  • Scenario: Another user deletes an item while you’re viewing the list.
  • Steps:
    1. Stay on a page with the item visible.
    2. Simulate another user deleting the item.
  • Expected Outcome:
    • The item is marked with a gray background and a “Deleted” badge.
    • A message appears: “An item has been deleted by another user.”
    • Pagination remains unchanged until refresh.

Test Cases for Concurrency

Test Case 14: Editing an Item That Was Modified by Another User

  • Scenario: Attempt to save changes to an item that was modified by another user after you opened the edit form.
  • Steps:
    1. Open the edit form for an item.
    2. Simulate another user modifying the same item.
    3. Attempt to save your changes.
  • Expected Outcome:
    • An error message is displayed: “This language has been modified by someone else.”

Test Case 15: Deleting an Item That Is Being Edited by Another User

  • Scenario: Attempt to delete an item that is currently being edited by another user.
  • Steps:
    1. Simulate another user opening the edit form for an item.
    2. Attempt to delete the same item.
  • Expected Outcome:
    • An error message is displayed: “Cannot delete language while it is being edited by another user.”
